- **2026-07-02** [Feature] **tddy-sandbox-qemu — QEMU VM sandbox backend (new crate)** — implements the `tddy-sandbox` `spawn_plan(SandboxPlan) -> SandboxHandle` contract over a booted QEMU VM: `qemu_sandbox_argv`/`ninep_fsdev_args`/`overlay_create_argv`/`guest_plan_json` (pure, unit-tested); `spawn_plan_with` creates an ephemeral qcow2 overlay, boots `qemu-system-x86_64` with one virtio-9p share per `MountSpec` plus reserved `tddy-runner`/`tddy-plan` shares, and polls the forwarded TCP control port for readiness; `run_sandbox_qemu` connects to the in-guest `tddy-sandbox-runner` (already TCP-capable) via the existing `connect_sandbox_client`/`run_host_relay`, streaming guest command output — no changes needed to shared darwin/cgroups sandbox infrastructure. `tddy-daemon` gains an opt-in `TDDY_SANDBOX_BACKEND=qemu` selector (existing backends remain default). Guest-side 9p + init-hook (`docs/guest-image-9p-init.md`, kernel Kconfig fragment, BusyBox init script) documented with real artifacts, verified via host-side `dash`/`jq` simulation but not yet exercised inside an actual booted guest. Known approximation: exit code is 0/1 (clean-end vs. failure), not the guest command's real exit code — `run_host_relay` doesn't yet surface it.
